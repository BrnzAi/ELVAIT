Technical and Organisational Measures

ELVAIT — elvait.ai · Annex to the Data Processing Agreement

Effective Date: 26 February 2026

Version: 1.0

Applies to: All personal data processed by ELVAIT on behalf of Controllers

Reference: Article 32 GDPR — Security of Processing

The following technical and organisational measures are implemented by BRNZ AI, operating as ELVAIT, to ensure a level of security appropriate to the risk of processing personal data through the ELVAIT platform.

1. Encryption

1.1 Encryption in Transit

  • All connections to elvait.ai are encrypted using TLS 1.2 or higher.
  • HTTP requests are automatically redirected to HTTPS.
  • API endpoints enforce TLS; unencrypted connections are rejected.
  • Internal service-to-service communication within the cloud environment uses encrypted channels.

1.2 Encryption at Rest

  • All data stored in Google Cloud SQL (PostgreSQL) is encrypted at rest using AES-256 encryption, managed by Google Cloud's default encryption.
  • Database backups are encrypted using the same mechanism.
  • User passwords are hashed using industry-standard algorithms (bcrypt) and are never stored in plaintext.

2. Access Control

2.1 Authentication

  • Access to the platform requires authentication via email and password.
  • Passwords must meet minimum complexity requirements.
  • Session tokens are cryptographically generated and have defined expiration periods.
  • No IP addresses are collected or stored as part of standard platform operation.

2.2 Role-Based Access Control (RBAC)

  • The platform implements role-based access control, ensuring users can only access data and functions relevant to their role.
  • Survey participants can only access surveys they have been invited to.
  • Administrative functions are restricted to authorised Controller administrators.
  • Principle of least privilege is applied throughout the application.

2.3 Administrative Access

  • Access to production infrastructure is restricted to authorised BRNZ AI personnel.
  • Administrative access requires multi-factor authentication (MFA).
  • Access rights are reviewed periodically and revoked promptly upon role changes.

3. Database Security

  • The production database runs on Google Cloud SQL for PostgreSQL in the europe-west1 (Belgium) region.
  • The database instance is configured with a private IP address only — it is not accessible from the public internet.
  • Database connections are restricted to authorised application services within the Virtual Private Cloud (VPC).
  • Automated daily backups are enabled with a retention period of up to 30 days.
  • Point-in-time recovery is available for the backup retention window.
  • Database patches and updates are applied in accordance with Google Cloud's managed service schedule.

4. Pseudonymisation

  • Survey responses are stored linked to internal identifiers (IDs), not directly to personally identifiable information such as name or email.
  • The mapping between user identity and survey responses is maintained separately from response data where technically feasible.
  • Aggregated and anonymised reporting is used wherever possible to minimise exposure of individual data.

5. Logging and Monitoring

  • Application and infrastructure logs are maintained using Google Cloud's logging services.
  • Logs capture access events, errors, and security-relevant actions.
  • Logs do not contain plaintext passwords or full personal data.
  • Monitoring alerts are configured for anomalous activity, error spikes, and resource thresholds.
  • Log retention follows Google Cloud defaults and operational requirements.

6. Physical Security

  • All infrastructure is hosted in Google Cloud data centres, which maintain comprehensive physical security measures including:
  • 24/7 on-site security personnel and video surveillance;
  • Biometric and multi-factor access controls;
  • Redundant power supplies and environmental controls;
  • Compliance with ISO 27001, SOC 2 Type II, and other industry certifications.
  • For details, see Google Cloud Security.

7. Incident Response

  • BRNZ AI maintains an incident response process for security incidents affecting personal data.
  • Upon detection of a personal data breach, the following steps are taken:
    • Identification and containment — immediate assessment of scope and impact;
    • Notification — the affected Controller is notified within 72 hours as per the DPA;
    • Investigation — root cause analysis is conducted;
    • Remediation — corrective measures are implemented to prevent recurrence;
    • Documentation — all incidents and responses are documented.
  • Contact for security incidents: the email address provided in the ELVAIT Terms of Service.

8. Availability and Resilience

  • The Service is deployed on Google Cloud infrastructure with built-in redundancy.
  • Automated database backups ensure data can be restored in the event of failure.
  • The application is designed for rapid recovery and redeployment.

9. Regular Security Reviews

  • Technical and organisational measures are reviewed at least annually and updated as necessary to reflect changes in technology, threats, or the Service.
  • Dependency and vulnerability scanning is performed as part of the development lifecycle.
  • Infrastructure configurations are reviewed for security best practices.
  • This document is updated to reflect any material changes to security measures.

10. Data Minimisation

  • The platform collects only the personal data necessary to provide the Service: business email, name, and survey responses.
  • No IP addresses are collected or stored.
  • No special categories of personal data (Article 9 GDPR) are processed.
  • Data retention is limited to the duration of the Controller's subscription, with deletion upon request or termination.
